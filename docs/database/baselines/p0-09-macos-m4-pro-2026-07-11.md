# P0-09 macOS 基线（2026-07-11）

本报告冻结 Phase 0 的第一个可重复基线。它是当前实现的回归起点，不是跨平台性能承诺，也不把 SQL microbenchmark 当成完整页面打开耗时。

## 环境与方法

- 设备：MacBook Pro `Mac16,8`，Apple M4 Pro（12 核：8P+4E），24 GB RAM。
- 系统：macOS 26.5.2（25F84）。
- Flutter：3.44.4 stable；Dart 3.12.2 arm64。
- 固定 seed：`20260711`。
- 数据生成：`dart run tool/chat_database_v2_benchmark.dart --output=<临时目录>`。
- UI profile：`flutter drive --driver=test_driver/integration_test.dart --target=integration_test/p0_performance_baseline_test.dart -d macos --profile`。
- SQL 指标为同一进程只读 SQLite 查询 30 次的 p50/p95；单位为微秒。fixture 生成 WAL 是生成器单事务的压力值，不等于正常聊天流式 WAL。
- UI profile 是隔离 benchmark surface：D4 以 32 个 checkpoint 构造完整 1 MiB Markdown（代码、表格、Mermaid、图片标记各一段，其余为中英文普通文本）；D5 使用 100 个 3840×2160 PNG、1,000 行表格与 10,000 行代码并程序化滚动。
- DB2-02 execution-isolate probe 使用生产 `AppDatabase.open` / `NativeDatabase.createInBackground` 通路，在同一 SQLite connection 注册 direct-only 标量函数，并由递归 CTE 调用 64 次。函数回调发生在实际 SQLite worker isolate，计数 opening/root isolate 与 background isolate；profile test 本身运行在 Flutter UI isolate。

UI Runner RSS：开始 148,684,800 bytes，采样峰值 1,673,576,448 bytes，结束 1,614,921,728 bytes。D4 明显不满足候选 frame/RSS SLO，证明长流式 Markdown 仍是后续 renderer/timeline 的高优先级问题；不能因为 P0-03 已减少数据库写入就宣称输出渲染不卡。

第一次 profile harness 将代码/表格/Mermaid 复合块重复填满 1 MiB，Runner 在返回数据前退出（driver `Service has disappeared`）。这次失败未计为通过数据；fixture 随后修正为“每类结构各一段 + 普通文本填充到 1 MiB”，仍保留完整大小并得到上表可重复结果。

## DB2-02 execution-isolate profile 补证

同一 macOS profile harness 在 DB2-02 补证运行中返回：

| samples | opening/UI isolate calls | background isolate calls | 结果 |
| ---: | ---: | ---: | --- |
| 64 | 0 | 64 | `PASS` |

这证明 live `AppDatabase` 的 SQLite 引擎回调不在打开数据库的 Flutter UI isolate 执行，并与删除同步 `_syncDb`/repository API 的静态审计互相印证。本探针不覆盖 restore/backup/inspection 的显式 maintenance raw SQLite API；这些路径的调用线程、RSS 与阻塞预算继续由 OPS-02/发布门禁单独验证。

本次补证复跑同时得到 D4 build+raster p95 731.361 ms、31/33 超过 16.7 ms，D5 build+raster p95 3.623 ms、0/26 超过 16.7 ms；结论与冻结基线一致，因此不替换上表原始数值。
